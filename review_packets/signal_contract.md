# Signal Contract Design

## Phase 3 — Execution Signal Contract

### Objective
Define a strict and deterministic execution signal structure for downstream execution systems, brokers, strategy engines, and trading automation pipelines.

The signal contract standardizes:
- prediction output
- execution decisions
- confidence reporting
- timestamp tracking
- request traceability

---

# Signal Contract Structure

## Required Fields

| Field | Type | Description |
|---|---|---|
| symbol | string | Trading symbol/ticker |
| action | string | Trading action: BUY / SELL / HOLD |
| confidence | float | Confidence score between 0 and 1 |
| predicted_price | float | Model predicted future price |
| timestamp | string (ISO-8601) | UTC timestamp of signal generation |
| request_id | string | Unique API request identifier |

# Signal Generation Rules

## Rule 1 — Deterministic Output

Identical input must produce identical execution signal.

---

## Rule 2 — Timezone Standardization

All timestamps must:
- use UTC
- use ISO-8601 format
- avoid tz-aware/tz-naive conflicts

---

## Rule 3 — Confidence Constraints

Confidence must:
- remain between 0 and 1
- be deterministic for identical input
- not contain stochastic variation

---

## Rule 4 — Action Safety

Invalid actions are prohibited.

Allowed values only:
```text
BUY
SELL
HOLD
```
